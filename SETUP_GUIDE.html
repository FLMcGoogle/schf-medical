Firebase Setup Guide

Space Coast Honor Flight — One-time backend configuration (~45 minutes)

📋 Steps in This Guide

  1. Create Your Firebase Project
  2. Enable Firestore Database
  3. Set Security Rules
  4. Enable Authentication & Create Provider Accounts
  5. Get Your Config Snippet
  6. Paste Config into Both HTML Files
  7. Host on Firebase Hosting (Free)
  8. Test End-to-End
  9. Day-of-Flight Usage
1

Create Your Firebase Project

🖥 Go to

Open a browser and go to: firebase.google.com — sign in with any Google account.

Click "Go to console" in the top right, then click "Add project". Name it something like schf-medical. Disable Google Analytics when asked (not needed). Click Create project. Wait about 30 seconds while Firebase sets it up, then click Continue.

✓ Done when: You see the Firebase project dashboard with a sidebar on the left showing "Build", "Run", "Analytics" sections.
2

Enable Firestore Database

In the left sidebar, click Build → Firestore Database. Click "Create database". Choose "Start in production mode" (we'll set proper rules in Step 3). For location, choose us-east1 (closest to Florida). Click Enable.

✓ Done when: You see an empty Firestore panel that says "Start a collection".

You don't need to create any collections manually — the app creates them automatically when the first veteran submits a form.

3

Set Security Rules

In Firestore, click the "Rules" tab at the top. You'll see a rules editor. Delete everything currently in there and paste in the rules below exactly:

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Veterans collection
    match /veterans/{veteranId} {

      // Anyone can CREATE a new record (veteran submitting from home)
      allow create: if true;

      // Only authenticated providers can READ, UPDATE, or DELETE
      allow read, update, delete: if request.auth != null;
    }
  }
}

Click Publish. These rules mean: veterans can submit their forms without logging in, but only your authenticated providers can see and edit the data.

⚠️ Important: Do not skip this step. Without these rules, the database defaults to blocking ALL access, and nothing will work.
4

Enable Authentication & Create Provider Accounts

In the left sidebar, click Build → Authentication. Click "Get started". Under "Sign-in method", click Email/Password, toggle it to Enabled, and click Save.

Now click the "Users" tab. Click "Add user". Enter the email and password for each provider who will use the dashboard. Repeat for each person. You can add or remove accounts here at any time.

📝 Suggested accounts to create

One account per provider/medical staff member who will access the dashboard. Use real email addresses — they'll use these to log in. Example: drsmith@spacecoasthonorflight.org

✓ Done when: You can see your providers listed in the Users tab with their email addresses.
5

Get Your Config Snippet

Click the gear icon ⚙️ next to "Project Overview" in the left sidebar. Choose "Project settings". Scroll down to the section called "Your apps". If there are no apps yet, click the </> Web icon to register a web app. Give it any nickname like "SCHF Web". Click Register app.

You'll see a block of JavaScript that looks like this — this is your config. Copy it all.

// Your config will look exactly like this, with real values:
const firebaseConfig = {
  apiKey:            "AIzaSyBxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  authDomain:        "schf-medical.firebaseapp.com",
  projectId:         "schf-medical",
  storageBucket:     "schf-medical.appspot.com",
  messagingSenderId: "123456789012",
  appId:             "1:123456789012:web:abc123def456"
};
Note: The apiKey here is NOT a secret — it's safe to put in your HTML files. Firebase security is controlled by the Rules you set in Step 3, not by hiding this key.
6

Paste Config into Both HTML Files

Open both HTML files in a text editor (Notepad, TextEdit, VS Code, etc.). In each file, find the block that looks like this and replace the placeholder values with your real values from Step 5:

veteran_intake_v2.html

The form veterans fill out at home. Needs the config so submissions go to your database.

provider_dashboard_v2.html

The provider portal. Needs the same config to read/write the same database.

In each file, find this section near the top and replace the 6 placeholder values:

// FIND THIS (in both files):

  apiKey:            "PASTE_YOUR_API_KEY_HERE",
  authDomain:        "PASTE_YOUR_PROJECT_ID.firebaseapp.com",
  projectId:         "PASTE_YOUR_PROJECT_ID",
  storageBucket:     "PASTE_YOUR_PROJECT_ID.appspot.com",
  messagingSenderId: "PASTE_YOUR_SENDER_ID",
  appId:             "PASTE_YOUR_APP_ID"
};

// REPLACE WITH YOUR REAL VALUES (same in both files):
const firebaseConfig = {
  apiKey:            "AIzaSyBxxxxxxxxxxxxxxx",
  authDomain:        "schf-medical.firebaseapp.com",
  projectId:         "schf-medical",

⚠️ Both files must have the same config so they connect to the same database. If they have different project IDs, they won't share data.
7

Host on Firebase Hosting (Free)

The easiest way to host is Firebase Hosting — it's free, fast, and gives you permanent URLs to share. This requires installing Node.js and the Firebase CLI once.

Option A — Firebase Hosting (Recommended)

1. Install Node.js from nodejs.org (LTS version).
2. Open Terminal (Mac) or Command Prompt (Windows). Run: npm install -g firebase-tools
3. Run firebase login and sign in with your Google account.
4. Create a folder, put both HTML files inside it, rename them to index.html (veteran form) and provider.html (dashboard).
5. In that folder, run firebase init hosting — say yes to existing project, use current directory as public folder, say no to single-page app.
6. Run firebase deploy.
Your URLs will be: https://schf-medical.web.app/ (veteran form) and https://schf-medical.web.app/provider.html (dashboard).

Option B — GitHub Pages (Simpler if you have a GitHub account)

Create a new public repository on GitHub. Upload both HTML files. Enable Pages under Settings → Pages → main branch. Your URLs will be https://yourusername.github.io/repo-name/veteran_intake_v2.html and similar. This works perfectly since both files already have Firebase embedded.

Option C — Any Web Host

Since everything is self-contained in the HTML files, you can upload them to ANY web host — GoDaddy, Squarespace files section, your organization's intranet, Dropbox public link, etc. The files work anywhere because Firebase is loaded from Google's CDN.

8

Test End-to-End

☐Open the veteran intake form. Fill it out completely and submit. You should see the "Thank You" success screen.
☐Open the Firebase console → Firestore Database. You should see a veterans collection appear with the submission as a document.
☐Open the provider dashboard. Log in with an account you created in Step 4.
☐The submitted veteran record should appear on the Roster tab.
☐Tap the veteran card to expand it. Assign a Patient # and toggle a flag. It should save instantly (green "Saved to cloud" flash).
☐Open the provider dashboard on a different device or browser. The same data should appear — confirming live sync is working.
9

Day-of-Flight Usage

Before the flight: Providers open the dashboard, use the Sort by "Flight Date" to see that day's manifest. All veteran records from their home submissions plus any interview notes are already there. Patient numbers assigned during interviews will sort them into manifest order.

On the flight: Providers bookmark provider.html on their phones. No download needed — it works in any mobile browser. As long as the phone has cell signal or WiFi, all changes sync in real time across all provider phones simultaneously. Tap any veteran to see their full medical record and phone number.

If there's no internet on the plane: The most recently loaded data will still be visible (cached by the browser). Changes made offline will sync when connectivity returns.

Pro tip: Have each provider "Add to Home Screen" from their phone browser before the flight day. This makes it launch full-screen like a native app, with faster load times.
